# High-Level Architecture

## System Overview

The platform is a **real-time AI-powered communication system** supporting multiple conversation modes with integrated language skill evaluation. The architecture follows an **event-driven microservices** pattern with WebSocket-based real-time communication.

## Conversation Mode Flows

### Mode 1: 1-to-1 AI Conversation
```
User в”Җв”Җв–ә WebSocket в”Җв”Җв–ә Session Service в”Җв”Җв–ә AI Service в”Җв”Җв–ә LLM API
  в–І                                            в”Ӯ
  в”Ӯ                                            в–ј
  в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җ TTS в—„в”Җв”Җв”Җв”Җ Response в—„в”Җв”Җв”Җв”Җ Eval Service (scoring)
```
- User speaks вҶ’ STT converts to text вҶ’ LLM responds вҶ’ TTS speaks back
- Every exchange is evaluated for speaking/listening/writing quality

### Mode 2: 1-to-1 Peer Conversation (AI Silent Monitor)
```
User A в”Җв”Җв–ә WebSocket в”Җв”Җв–ә Session Service в”Җв”Җв–ә User B
                              в”Ӯ
                              в–ј (async, non-blocking)
                         AI Service (silent analysis)
                              в”Ӯ
                              в–ј
                         Eval Service (post-session scoring)
```
- AI receives the full transcript stream but does NOT inject messages
- Scoring happens in real-time background or post-session

## Key Design Principles

| Principle | Implementation |
|-----------|---------------|
| **Separation of concerns** | Each service owns its domain completely |
| **Event-driven** | Services communicate via Redis Pub/Sub events |
| **Stateless services** | All state in PostgreSQL/Redis, services are horizontally scalable |
| **Graceful degradation** | If AI service is slow, conversations continue; scoring queues |
| **Privacy by design** | Listeners are anonymous; speaker data is scoped to sessions |
| **Idempotent operations** | All API calls and event handlers are idempotent |
